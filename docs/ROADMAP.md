# CreatorConnect — Build Roadmap

*Sequencing document. Does not redesign product or architecture — implements `PRODUCT.md` and `ARCHITECTURE.md` in order. Written for a solo founder who already knows SvelteKit + Supabase + Stripe Connect from Lota; estimates assume that experience, not a green engineer or a team.*

**Revision note (July 2026, following PRODUCT.md's and ARCHITECTURE.md's Phase 0 revisions):** Phase 0's creator/manager interviews (0.1/0.2 below) are now complete, and the headline finding — creators want to choose their pricing mechanism per listing (A, C, or D) rather than have one imposed — is a real scope increase, not a reframing. This revision reworks **Phase 1's epic list** (Section 3) to build three mechanism-specific flows instead of one, makes an explicit sequencing call (ship D first, A/C as a fast-follow — see Section 3), updates **Phase 2.1's test-suite scope** and **Phase 1's time estimate** (Section 7) to account for ARCHITECTURE.md's new risk #6 (tripled RLS/RPC surface area), and adds a note to **Section 6's kill-switch signals**. Phase 0's other items (0.3-0.6), the tiebreaker-deferral judgment call, and the tension-flagged section are otherwise unaffected and preserved as-is, with one added cross-reference note in the tension section.

## 1. Phasing philosophy

The core logic is **validate the two cheapest-to-check unknowns before writing marketplace code, then build the smallest correct end-to-end loop, then harden, then expand** — not "build everything, then see if anyone shows up." Two things gate engineering start: (a) the creator-interview track from PRODUCT.md Section 7, because several of those questions (deposit tolerance, dispute shape, manager delegation model) directly determine what Phase 1 should even build, not just how well it's received; and (b) a narrow Stripe Connect delayed-transfer check, because ARCHITECTURE.md Section 8 #4 flags the entire escrow design as unverified against Stripe's actual current docs — if that assumption is wrong, it changes schema and RPC design, not just a config value. Both are cheap (days, not weeks) and both block real engineering decisions, so they come first and are sequential-ish with engineering, not parallel to it. Everything downstream of Phase 0, however, does **not** wait on more interviews — PRODUCT.md is explicit that some open questions (deposit %, response-window length, manual-entry staleness tolerance, dispute volume) can only be answered with real usage data, not more talking. So the interview track continues in parallel through every later phase as an ongoing signal-gathering process, not a phase-0-only checkbox, while engineering proceeds on the founder's best current judgment once the two blocking Phase 0 questions are answered.

## 3. Phase 1 — Core reservation loop (MVP)

**Goal: the smallest real slice that proves creator lists → advertiser reserves/offers/requests-exclusivity → creator confirms → escrow moves → delivery → payout, for the pilot cohort from 0.3.**

**Sequencing call — ship mechanism D first as the full Phase 1 slice; build A and C as a fast-follow immediately after, not simultaneously with D. Stated explicitly, not assumed.** The alternative (build all three mechanisms in one Phase 1 slice before shipping anything) was considered and rejected, for three reasons specific to where this project actually is:
- **Interviews validated D standalone; A/C were an additional ask, not a rejection of D.** 0.1's finding is "creators are fine with D, *and* some also want A/C as options" — not "D doesn't work without A/C." Nothing about D's value proposition depends on A/C existing alongside it on day one.
- **D is already fully designed and is the richer, harder mechanism** (deposit, escrow-on-reservation, sealed-bid tiebreaker contention handling, concurrency locking per risk #2). A and C are both structurally simpler (no deposit, no tiebreaker — ARCHITECTURE.md Section 2). Shipping the hard, already-fully-specified mechanism first and proving the whole loop (listing → escrow → payout) end-to-end derisks the parts of the system every mechanism shares (`deals`, escrow, contract gen, delegation) before spending time on two more listing/negotiation UIs.
- **The founder is solo, and risk #6 explicitly recommends this sequencing at the engineering-practice level**: "build and test mechanism D's RPC family first... then treat A and C's RPC families as requiring the same test rigor from scratch rather than assuming 'D worked, so the pattern is proven.'" Building all three in parallel in one slice would fight that advice by forcing context-switching across three RPC families and three RLS surfaces before any one of them has been proven correct under real usage.

Concretely: **Phase 1 below ships mechanism D only** (matching the original single-mechanism MVP almost exactly, with the addition of the `pricing_mechanism` field and mechanism-selection step so the schema/UI don't need to be retrofitted later). **Phase 1-FastFollow (new, immediately after Phase 1, before Phase 2's hardening work) ships A and C.** This means the pilot cohort's first real listings will initially only offer D — acceptable because 0.1 confirmed D works fine on its own, and the pilot cohort was recruited around D-style interviews to begin with (0.3). If pilot feedback in the first weeks of Phase 1 strongly signals creators are blocked without A/C (not expected, but worth watching), that would be a reason to pull Fast-Follow forward — treat it as a signal to monitor, not a default assumption.

### Phase 1 (D only) — 8 epics, sequenced

1. **Reservation concurrency fix, built and load-tested first (ARCHITECTURE.md #2).** Build `place_reservation` as a `security definer` RPC with a `select ... for update` row lock (or advisory lock) on the listing, before any other reservation-flow code exists. Load-test with concurrent simulated requests against a single listing. **Done when:** two simultaneous reservation attempts against the same listing reliably produce exactly one `held` reservation and one rejected/redirected-to-tiebreak attempt, verified under actual concurrent load, not just sequential test cases. This is explicitly first — ARCHITECTURE.md flags it as the failure mode that "won't show up until real contention happens live," and it's the one bug class that directly breaks user trust (double-booking a paid deposit). **Unchanged by the three-mechanism revision** — this is D-specific, and ARCHITECTURE.md risk #6 explicitly names it as the thing to prove first before A/C get built.
2. **Auth/role scaffolding + RLS foundation.** `profiles` table, three roles, `public_profiles` view for browse, base RLS policies, the `is_authorized_for_creator` helper function (used by every later table, across all three mechanisms once Fast-Follow lands). **Done when:** a creator, advertiser, and manager can sign up, get the right role, and RLS blocks cross-role reads/writes on `profiles`.
3. **Listing lifecycle + mechanism selection.** `creator_listings` table + RLS + manual-entry creation/edit UI (platform, content type, availability window, manual perf stats/demographics, constraints text), **plus the new `pricing_mechanism` field and its check-constrained mechanism-specific columns (ARCHITECTURE.md Section 2)**. **Updated scope:** even though only D is buildable end-to-end in this phase, the listing form and schema should support the mechanism-selection step now (creator picks A/C/D, sees the explainer copy per PRODUCT.md Flow 1) — for A and C, selecting them at this stage should either be hidden/disabled with a "coming soon" state or simply not offered as choices until Fast-Follow ships, whichever is less UI work; the `pricing_mechanism` column and its `check` constraint should exist from the start so Fast-Follow doesn't require a listings-table migration touching live data. Status machine `draft → open → reserved/expired/cancelled` (D's machine), with the "only RPCs move status past open" rule enforced by revoking direct table grants. **Done when:** a pilot creator can publish a real D listing and it's visible in browse, with the mechanism clearly labeled.
4. **Browse/filter/shortlist for advertisers.** Filter by platform, content type, follower range, price range, availability window. Each listing shows its pricing mechanism label (PRODUCT.md Flow 2) — trivial to add now even though only D listings exist yet, since it avoids a later UI change when A/C listings start appearing. Shortlist/watch with no commitment. **Done when:** an advertiser can find a pilot creator's listing and shortlist it without any payment step.
5. **Reservation + deposit flow (built on top of epic 1's lock).** `reservations` table, Stripe PaymentIntent for the 10% deposit on the platform account, webhook-driven `escrow_transactions` mirroring (never client-trusted), confirmation-deadline countdown, auto-expire + refund via the outbox/pg_cron pattern. **Done when:** an advertiser can place a real deposit in test mode, the slot leaves `open`, and expiry/refund fires correctly if the creator doesn't respond in the window.
6. **Creator confirmation → contract → escrow completion.** `confirm_deal_as` RPC (price at/above floor, band-checked for delegated managers), `deals` row creation with FTC disclosure boilerplate baked in and a generated contract PDF, remaining-balance charge (charge 100% up front per ARCHITECTURE.md Section 4's recommendation), first Transfer tranche. **Mechanism-agnostic by design, noted explicitly:** `deals`, contract generation, and the escrow/Stripe sequence are the convergence point ARCHITECTURE.md Section 2/4 describes — they work identically regardless of which mechanism produced the row. Building this against D now means Fast-Follow's `accept_offer_as`/`convert_exclusivity_as` (epic 9 below) plug into the same contract/escrow code unchanged, just with a different RPC populating `deals.offer_id`/`exclusivity_grant_id` instead of `reservation_id`. **Done when:** a confirmed deal produces a real contract artifact and moves real (test-mode) money in the first tranche.
7. **Delivery sign-off, auto-release, and basic dispute flag.** Advertiser sign-off or 5-day auto-release timeout triggers `release_delivery_balance`; either party can flip `deals.status = 'disputed'` to freeze it (manual founder resolution via direct DB/Stripe access — no admin UI yet, per ARCHITECTURE.md #5's explicit MVP deferral). **Mechanism-agnostic, same convergence-point reasoning as epic 6** — nothing here needs to know which mechanism produced the deal. **Done when:** the full loop closes on a test deal — deposit, confirmation, delivery, payout minus platform fee — with dispute-freeze provably blocking the auto-release job.
8. **Manager POA — minimum viable delegation for mechanism D (not the full Phase 2 hardening, and not yet A/C's band-checks).** Just enough of ARCHITECTURE.md Section 3 to onboard the 1-2 pilot managers from 0.3: `manager_creator_links` (creator-initiated approval, creator-only revoke), `listing_price_bands`, the `create_listing_as`/`confirm_deal_as` RPCs with band-check, and `audit_log` writes in the same transaction. **Includes the onboarding/explainer copy for the band-based auto-accept model** that 0.2's interview finding flagged as needed (managers understood and liked the model once explained — this is a copy/UX task folded into this epic, not new engineering). **Scope note:** this epic covers D's band-check only (`confirm_deal_as`); Fast-Follow adds the equivalent band-checks for A (`accept_offer_as`/`counter_offer_as`) and C (`convert_exclusivity_as`) per ARCHITECTURE.md Section 3's "three parallel RPC families" design — building all three now would mean band-checking RPCs for mechanisms that don't have listings yet. **Done when:** a pilot manager can list and confirm within-band D deals on behalf of a linked creator, with a visible audit trail the creator can inspect and clear explainer copy shown during onboarding. Full cross-mechanism RLS-bypass test suite is Phase 2 (2.1), not here — but D's band-check logic itself must work correctly from day one since it's directly handling money, and should be tested to the rigor ARCHITECTURE.md risk #6 calls for as the "test D first" baseline the A/C epic will be held to.

### Phase 1-FastFollow (new) — mechanisms A and C, immediately after Phase 1 ships

**One combined epic for A and C, not two separate epics — a judgment call, justified:** A and C are both structurally lighter than D (no deposit, no escrow-on-hold step, no tiebreaker — ARCHITECTURE.md Section 2 explicitly describes both as "lighter-weight" than `reservations`), they share the same shape of change (one new table, a small RPC family, a UI branch off the existing mechanism-selection step already built in Phase 1 epic 3), and — most importantly — by the time this epic starts, the founder will have just finished proving out the harder D pattern (RPC families, RLS, audit logging, band-checks) end-to-end, so A and C are applications of an already-validated pattern rather than novel design work. Splitting them into two epics would mostly just split a single "apply the proven pattern twice, to two small tables" effort into two milestones with no real independent risk boundary between them. (If, once underway, C's exclusivity-window/expiry-job piece turns out to carry more real complexity than expected — e.g., because of how it interacts with the browse/discovery UI — it's fine to split it out mid-epic; this is a starting judgment call, not a hard constraint.)

9. **Mechanisms A + C: lighter-weight pricing flows.**
   - **Schema:** `listing_offers` (mechanism A) and `listing_exclusivity_grants` (mechanism C) tables + RLS, per ARCHITECTURE.md Section 2. Extend `deals`' origin check constraint usage (`offer_id`/`exclusivity_grant_id` — already added to the schema in epic 6 above) with real writers.
   - **RPCs:** `accept_offer_as`/`counter_offer_as`/`reject_offer_as` for A; `request_exclusivity_as`/`convert_exclusivity_as` for C — each band-checking against `listing_price_bands` via the shared `check_price_band` helper (ARCHITECTURE.md Section 3), same pattern as D's `confirm_deal_as` from epic 8, applied fresh per risk #6's guidance rather than assumed safe by analogy.
   - **UI:** sequential accept/reject/counter thread view for A; exclusivity-request + bilateral-negotiation-once-converted view for C. Both listing types get full mechanism-selection support in the epic-3 creation flow (removing the "coming soon" state).
   - **Background job:** C's own expiry-scan pg_cron job (`listing_exclusivity_grants` where `status = 'active' and window_ends_at < now()`, ARCHITECTURE.md Section 5) — distinct from D's job, no Stripe/refund step since C never took a deposit. A has no deadline-driven job, by design (nothing to build here for A).
   - **Realtime:** subscriptions on `listing_offers` and `listing_exclusivity_grants` alongside the existing `reservations` subscription (ARCHITECTURE.md Section 6) — additive, not a new pattern.
   - **Testing:** per ARCHITECTURE.md risk #6's explicit instruction, treat A and C's RPC families as needing the same RLS-bypass test rigor as D's from epic 1/8, from scratch — not assumed safe because D worked. This epic's "done" bar includes basic band-check and RLS coverage for A and C individually; the *full* cross-mechanism suite (all three families tested together, including interactions) is Phase 2.1.
   - **Done when:** a pilot creator can publish an A or C listing, an advertiser can complete the full A negotiation thread or C exclusivity-then-conversion flow, and a confirmed deal from either flows into the same contract/escrow/delivery pipeline built in Phase 1 epics 6-7 with no changes needed there.

**Explicit judgment call — the sealed-bid tiebreaker is deferred to Phase 1.5, not built in MVP.** PRODUCT.md Section 6 lists the tiebreaker as in-scope, but also says contention will be rare, and Section 7 Q1 says the tiebreaker specifically needs 10-15 creator interviews before committing engineering time to it. Building `deal_tiebreaks`/`tiebreak_bids`, the sealed-read RLS, and the resolution job for a scenario that (a) may not occur at all in a pilot cohort of 5-8 creators with modest concurrent advertiser interest, and (b) PRODUCT.md itself flags as unvalidated, is premature engineering investment against near-zero users. Instead: epic 5's `place_reservation` RPC should detect the contention case (a second reservation attempt against a `held` listing) and simply reject it with a "slot no longer available" response for MVP — correct and safe, just not fair-optimal. Build the actual tiebreaker in **Phase 1.5**, triggered the first time real contention is observed in pilot usage (or proactively once 2-3 pilot listings show shortlist/watch activity from multiple advertisers, as an early signal it's about to happen). This keeps the concurrency-safety property (epic 1) intact — no double-booking — while deferring the fairness mechanism until there's a real instance to design against instead of a hypothetical one.

## 4. Phase 2 — Manager/POA hardening + operational maturity

**Goal: the parts of the architecture that are correct to defer past a 5-8-creator pilot but not past real growth.**

- **2.1 — Full cross-mechanism delegation RLS/RPC test suite (ARCHITECTURE.md #1 and #6).** Formalize the "can a manager, via any code path, exceed their band or touch a creator they're not linked to" test suite ARCHITECTURE.md calls for explicitly — **and, per risk #6, this now means a test-suite section per mechanism, not one shared test that happens to exercise D.** Concretely: explicit coverage for `confirm_deal_as` (D, baseline from Phase 1 epic 8), `accept_offer_as`/`counter_offer_as` (A), and `convert_exclusivity_as` (C, from Phase 1-FastFollow epic 9) — each tested independently for band-bypass and cross-creator-access holes, plus the shared `check_price_band` helper tested once, thoroughly, on its own. Also verify the `deals` three-way-exclusive-origin `check` constraint (`reservation_id`/`offer_id`/`exclusivity_grant_id`) can't be violated by any of the three RPC families. Run the full suite on every migration going forward, not as a one-time check. This is the highest-priority Phase 2 item — ARCHITECTURE.md calls the (now-tripled) delegation surface area "the single biggest build risk," Phase 1/Fast-Follow shipped per-mechanism minimums as each mechanism landed; this is where they get tested together and to full rigor before manager count grows past the pilot accounts.
- **2.2 — Manager permission model revisit, if 0.2 interviews flagged a mismatch.** If the band-based model didn't match how pilot managers actually wanted to operate, this is where that gets addressed — informed by real pilot manager behavior on top of the original interview signal.
- **2.3 — Sealed-bid tiebreaker (moved up from "Phase 1.5" if not already built reactively).** If real contention hadn't occurred by end of Phase 1, and the platform now has enough listings/advertisers that it's likely to, build it proactively here rather than reactively.
- **2.4 — Notification/email polish.** Beyond the bare confirmation-deadline and dispute emails needed for Phase 1 to function: reminder emails at ~75% of confirmation window elapsed (per ARCHITECTURE.md Section 6), richer deal-lifecycle emails, in-app notification bell backed by the `notifications` table and Realtime.
- **2.5 — Minimal admin dispute UI (ARCHITECTURE.md #5).** Once dispute volume exceeds "a few, handled by hand via direct SQL/Stripe dashboard" — build the `/admin/disputes` route ARCHITECTURE.md describes: service-role-gated, surfaces `deals`/`escrow_transactions` state, lets the founder resolve (release or partial refund) without raw SQL. Trigger: not a fixed date — trigger is dispute volume actually becoming annoying to handle manually, per PRODUCT.md Section 7 Q4's own framing ("worth pressure-testing... before assuming it scales").
- **2.6 — Deposit % / response-window tuning.** Revisit the placeholder 10% / 24-48h figures (PRODUCT.md Section 7 Q2) using actual Phase 1 pilot data — conversion rate at different implied thresholds, response-window blow-through rate.

## 5. Phase 3 — Post-MVP expansion (only after real usage data)

Everything PRODUCT.md Section 6 explicitly deferred. Not re-justified here — just sequenced with its trigger condition.

- **Automated metrics ingestion (YouTube/IG/TikTok API pulls).** Trigger: manual-entry staleness is actually observed as a problem (PRODUCT.md Section 7 Q3) — listings going stale or advertisers citing distrust of self-reported stats as a reason not to reserve. Note: if this eventually requires scraping/headless-session workarounds rather than clean API polling, ARCHITECTURE.md Section 7 flags this as the one plausible future reason to stand up a Hetzner box (reusing the Lota pattern) — not needed until then.
- **AI-assisted/algorithmic price discovery.** Trigger: enough transaction data exists to train or justify it (PRODUCT.md Section 6) — realistically, dozens to low hundreds of confirmed deals with price history.
- **Secondary market/resale of reserved slots.** Trigger: not usage-driven — PRODUCT.md flags this as consumer-protection complexity not worth taking on before core liquidity exists. Revisit only after Phase 1/2 liquidity is proven and with counsel involved, given the regulatory framing in PRODUCT.md Section 1.
- **Manager bulk tools (bulk listing/rate-card edits across roster).** Trigger: a manager's roster size and Phase 1/2 per-creator tooling friction actually shows up as a retention risk — this is an efficiency feature for managers already succeeding on the platform, not a bootstrapping one.
- **Premium subscriptions.** Trigger: none from usage data — PRODUCT.md is explicit the transaction fee alone is sufficient to validate the core loop. Revisit as a revenue-diversification move only after the core loop is proven, not as a response to any specific pain signal.
- **Expansion beyond mid-tier creators / beyond the 4-8 week horizon.** Trigger: liquidity proven at the current segment and horizon (PRODUCT.md Section 2's explicit condition) — not before. Moving to celebrity/ambassador tier or long-tail creators, or to 3-6 month horizons, changes the core behavioral assumptions PRODUCT.md's mechanism is built around, so this isn't a simple scope-widening — treat it as a re-validation, not just an expansion.
- **International/multi-currency/multi-jurisdiction escrow.** Trigger: expansion beyond MVP geography is actually being pursued, and only after counsel review — not an engineering-led decision.

## 6. Cross-cutting: what runs in parallel with engineering, always

**Ongoing interview track (not a Phase 0 one-off).** PRODUCT.md Section 7's five questions aren't all answerable before launch — Q2 (deposit %/window), Q3 (staleness tolerance), and Q4 (dispute volume/shape) explicitly need real usage data, which only exists once Phase 1 is live. So: keep a lightweight, recurring cadence of creator/manager check-ins running through Phase 1 and Phase 2 (a standing "talk to 2-3 pilot users every 2 weeks" habit, not a formal research phase) — checking not just "does this still work" but actively re-testing the specific open questions against what's actually happening in the product. This is cheap relative to engineering time and is the only way Q2-Q4 actually get answered.

**Kill-switch signals — what would say mechanism D isn't working.** Watch these from the moment Phase 1 goes live with the pilot cohort:
- **Time-to-first-listing** is high or listings stall in `draft` — signals the manual-entry listing flow itself is too much friction for creators who already said yes in interviews (a gap between stated and revealed preference).
- **Time-to-first-reservation** on published listings is very long or zero across the pilot cohort — signals either advertiser discovery isn't working or the deposit ask is a bigger barrier in practice than 0.1's interviews suggested.
- **Deposit-refund rate is high** (reservations frequently expiring unconfirmed rather than converting to deals) — signals creators are treating the confirmation window as optional/low-priority, undermining the "advertiser gets a real hold" value prop core to mechanism D.
- **Creators editing/abandoning listings before reservation** at a high rate — signals the "set your terms, decide on offers as they come in" framing (PRODUCT.md Section 3) isn't landing, and creators still feel like they're committing to a stale price.
- **Dispute rate spikes** or disputes cluster around a specific cause (e.g., delivery-spec ambiguity) — signals the contract-generation step (epic 6) isn't capturing enough detail, not just a Phase 2 admin-UI problem.
- **Any of the above showing up in the 5-8 pilot creators specifically who were interview-positive in 0.1** is the strongest signal — if people who said yes to the mechanism in conversation don't behave that way with real money on the table, that's a stated-vs-revealed-preference gap worth pausing on before recruiting a wider cohort, not powering through with more marketing.

None of these should trigger an immediate pivot off one data point (5-8 pilot creators is a small sample) — but a cluster of 2+ of these signals within the first pilot cohort is the threshold for treating it as "revisit the mechanism," not "keep building Phase 2."

**New signal added by this revision — which mechanism creators actually choose, once A/C exist (Phase 1-FastFollow onward).** PRODUCT.md Section 7 Q6 names this explicitly as a real, accepted-not-avoided tradeoff: offering three mechanisms risks fragmenting advertiser discovery/liquidity, and the mitigation is post-launch usage data, not more upfront design. This belongs in the kill-switch signal list, not just as a product open question, because it's watchable from the same pilot-cohort data the other signals above use. Once Fast-Follow ships:
- **Mechanism-choice distribution at listing creation** — if one mechanism (most likely D, given it's launched first and already interview-validated, or possibly A for its simplicity) dominates heavily, that's not itself a failure signal, but it's the trigger PRODUCT.md Section 7 Q6 names for considering narrowing the default choice set or steering new listings toward the dominant mechanism.
- **Advertiser confusion/drop-off correlated with mechanism-mixed browse results** (e.g., advertisers shortlisting across mechanisms but converting only on one) — would be the concrete evidence that fragmentation, not just choice, is actually hurting discovery, distinct from creators simply having a preference.
This is a call made explicitly: it's a real signal worth tracking from Fast-Follow onward, but it is not a Phase 1 (D-only) signal, since a single-mechanism launch can't yet produce mechanism-distribution data.

## 7. Rough time framing

Order-of-magnitude, not a schedule — solo founder, known stack (SvelteKit + Supabase + Stripe Connect, direct carryover from Lota experience).

- **Phase 0 (validation & setup): 2-4 weeks.** Interviews (0.1/0.2, now complete) ran concurrently with Stripe/Supabase setup (0.4-0.6); the bottleneck was interview scheduling, not the technical setup. Unchanged by this revision.

Budget real slack here rather than compressing it. ## Tension flagged between the two source documents

PRODUCT.md Section 6 lists the sealed-bid tiebreaker as in-scope for v1 ("Reservation + deposit flow... including the closed-set sealed-bid tiebreaker for the rare contention case"), while PRODUCT.md Section 7 Q1 says the tiebreaker specifically needs 10-15 creator interviews *before committing engineering time to it* — those two statements are in mild tension with each other inside PRODUCT.md itself (one lists it as required MVP scope, the other says don't build it yet without validation). This roadmap resolves that tension in favor of Section 7's caution: the judgment call in Section 3 above (defer to Phase 1.5, contention-triggered) follows Q1's explicit instruction rather than Section 6's scope list, on the grounds that a validation gate stated explicitly for one feature should override a scope list that predates the gate being called out. Flagging this rather than silently picking a side, since a re-read of PRODUCT.md's own Section 6 language could reasonably argue the tiebreaker was intended as hard MVP scope — if creator interviews in Phase 0 (0.1) come back strongly reassuring on deposit tolerance *and* directly address contention comfort, it would be reasonable to pull the tiebreaker back into Phase 1 rather than deferring it.

**Note added by this revision:** Phase 0's 0.1 interviews did come back strongly reassuring on deposit tolerance (Q1 resolved above), but did not specifically surface contention comfort as a topic, so the condition in the previous paragraph for pulling the tiebreaker back into Phase 1 isn't met — the tiebreaker stays deferred to Phase 1.5 as designed. This tension is orthogonal to the mechanism-sequencing decision elsewhere in this revision (D first, A/C as Fast-Follow): the tiebreaker question is specifically about *when to build D's own contention-resolution logic*, while the A/C sequencing question is about *when to build two entirely different mechanisms*. The two deferrals don't interact or compound — deferring the tiebreaker doesn't make deferring A/C any easier or harder, and vice versa — they just happen to both be "build the harder/rarer thing later" calls made independently, for independent reasons.
